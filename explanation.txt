# 📒 Complete Notes — `fetch` & API Calls in JavaScript

---

## 1. What is `fetch`?

`fetch` is a **built-in JavaScript function** that lets your code communicate with the internet.

```
Without fetch → your JS lives only on your computer
With fetch    → your JS can talk to any server in the world
```

Real world analogy:
> You are sitting in a room. `fetch` is your **telephone** to call anyone in the world and ask for information.

---

## 2. `fetch` Syntax — The Big Picture

```js
fetch(url, optionsObject)
```

`fetch` takes exactly **two parameters:**

| Parameter | What it is | Example |
|-----------|-----------|---------|
| `url` | Where to send the request | `"https://api.groq.com/..."` |
| `optionsObject` | How to send it | `{ method, headers, body }` |

---

## 3. The Options Object — Every Property Explained

```js
fetch("url", {
    method: "POST",
    headers: { },
    body: " "
})
```

Think of this object like **instructions to a post office:**

---

### 3.1 `method`

```js
method: "POST"
```

Tells the server **what you want to do.**

| Method | Meaning | When to use |
|--------|---------|-------------|
| `GET` | Just fetch/read data | Loading a webpage |
| `POST` | Send data to server | Sending a message to AI |

> Default is `GET` if you don't write it. Always write `POST` explicitly when sending data.

---

### 3.2 `headers`

```js
headers: {
    "Content-Type": "application/json",
    "Authorization": "Bearer YOUR_API_KEY"
}
```

Headers = **extra information written on the outside of your envelope** before anyone opens it.

Not the main content — just metadata about the request.

**`Content-Type: application/json`**
- Tells server → "the data I'm sending is in JSON format"
- Without it → server might misread your data
- Think of it like → **"Contents: Glass, Handle with care"** written on a parcel

**`Authorization: Bearer YOUR_KEY`**
- Proves your identity to the server
- Server checks this **before** even reading your message
- `Bearer` is a mandatory standard keyword before your key — always
- Think of it like → **showing your ID card at a security gate**

> Headers are checked FIRST before body. No valid Authorization = request rejected immediately.

---

### 3.3 `body`

```js
body: JSON.stringify({
    model: "llama3-8b-8192",
    messages: conversationHistory
})
```

Body = **the actual letter inside the envelope.** The real content you're sending.

**Why `JSON.stringify`?**

Internet only understands **text strings** — not JavaScript objects.

```js
// JS Object → internet CANNOT read this
{ model: "llama3-8b-8192", messages: [] }

// After JSON.stringify → internet CAN read this
'{"model":"llama3-8b-8192","messages":[]}'
```

`JSON.stringify` simply converts JS object → text string. That's its only job.

**What's inside body for Groq:**

| Property | What it is | Example |
|----------|-----------|---------|
| `model` | Which AI brain to use | `"llama3-8b-8192"` |
| `messages` | Full conversation history | `conversationHistory` array |

---

## 4. What is `messages` array?

Groq doesn't take just one message. It takes the **entire conversation** as an array so AI has context.

Each message is an object with two properties:

```js
{
    role: "user",         // who said it
    content: "hello!"     // what they said
}
```

| Role | Meaning |
|------|---------|
| `"user"` | Message typed by the human |
| `"assistant"` | Reply from the AI |
| `"system"` | Background instructions to AI |

A full conversation looks like:
```js
[
    { role: "system",    content: "You are a helpful assistant" },
    { role: "user",      content: "What is AI?" },
    { role: "assistant", content: "AI is..." },
    { role: "user",      content: "Give an example" },
]
```

This is why we maintain `conversationHistory` — we keep pushing messages into it so AI always remembers the full conversation.

---

## 5. `async` / `await` — Why we need it

API calls take **time** — internet is involved.

```
Without async/await → JS moves to next line immediately
                    → reply hasn't arrived yet
                    → you get undefined ❌

With async/await    → JS waits at that line
                    → until reply arrives
                    → then moves forward ✅
```

Rules:
- Function that contains `await` must be marked `async`
- `await` can only be used inside `async` functions
- Put `await` before anything that talks to the internet

---

## 6. Handling the Response

```js
const response = await fetch(url, options);
const data = await response.json();
return data.choices[0].message.content;
```

**Step by step:**

| Line | What happens | Analogy |
|------|-------------|---------|
| `await fetch(...)` | Sends request, waits for reply | Posting letter, waiting at mailbox |
| `response` | Raw reply from server | Receiving sealed envelope |
| `response.json()` | Opens and reads the reply | Opening envelope, reading letter |
| `data` | Full JS object with all info | The entire letter content |
| `data.choices[0]` | First reply option from AI | First paragraph of letter |
| `.message.content` | Actual text of the reply | The answer you were looking for |

---

## 7. Complete `callAI` function with every line explained

```js
async function callAI(userText) {

    // Send request to Groq server
    const response = await fetch("https://api.groq.com/openai/v1/chat/completions", {
        
        method: "POST",              // We are SENDING data
        
        headers: {
            "Content-Type": "application/json",  // Sending JSON format
            "Authorization": "Bearer YOUR_KEY"   // Our ID card
        },
        
        body: JSON.stringify({
            model: "llama3-8b-8192",         // LLaMA 3 AI model by Meta
            messages: conversationHistory     // Full conversation for context
        })
    });

    // Convert raw response to readable JS object
    const data = await response.json();

    // Extract just the text reply from the big response object
    return data.choices[0].message.content;
}
```

Let me break this down super clearly.

## What's happening overall

You're sending a request to an AI API (Groq here, not Anthropic). Think of it like sending a letter — you need to pack it in a specific format.

---

## The Big Picture Flow

```
User types something
    ↓
Add it to conversationHistory list
    ↓
Send the WHOLE list to the AI
    ↓
AI reads the full conversation
    ↓
AI replies
```

---

## conversationHistory — what is it?

It's just a list that looks like this:

```javascript
conversationHistory = [
  { role: "user",      content: "Hi!" },
  { role: "assistant", content: "Hello! How can I help?" },
  { role: "user",      content: "What is 2+2?" },
]
```

Every time someone talks, you **push** a new object into this list. You send the **whole list** every time so the AI remembers the full conversation.

---

## JSON.stringify — why?

The internet can only send **text**. Not JavaScript objects.

```javascript
// This is a JS object — internet can't send this
{ model: "llama", messages: [...] }

// JSON.stringify turns it into a string — now it can travel
'{"model":"llama","messages":[...]}'
```

Think of it like putting your object **into an envelope** (text form) to mail it.

---

## The body you're sending

```javascript
body: JSON.stringify({
  model: "llama-3.3-70b-versatile",  // WHICH AI to use
  messages: conversationHistory       // the full chat history list
})
```

So the API receives this as text, unwraps it, and sees your messages list.

---

## data.choices[0].message.content — why so nested?

The API sends back a response that looks like this:

```javascript
data = {
  choices: [              // a LIST of possible replies (usually just 1)
    {                     // choices[0] = first (only) reply
      message: {
        role: "assistant",
        content: "The answer is 4!"   // ← THIS is what you want
      }
    }
  ]
}
```

So you drill down:
```
data               → the whole response
  .choices         → the list of replies
    [0]            → grab the first reply
      .message     → the message object inside
        .content   → the actual text!
```

---

## Why `choices` and not just `message`?

That's just how **this specific API** (Groq/OpenAI format) designed their response. Different APIs return differently shaped data. Anthropic for example returns `data.content[0].text` instead. You just have to match whatever shape the API gives you.

---

## Full picture in one diagram

```
You build:          { model: "...", messages: [ list of chat ] }
stringify it:       turns into text string
send to API  →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→ API receives it
                                                    AI thinks
get response ←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←← API sends back
Read:               data.choices[0].message.content = AI's reply text
```

## 📄 HTML — Full Explanation

---

### Boilerplate (Head section)

```html
<!DOCTYPE html>
```
Tells browser → "this is a modern HTML5 file." Always first line.

```html
<meta charset="UTF-8">
```
Supports all characters including emojis and special symbols.

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0">
```
Makes your page look correct on mobile screens. Without this → page looks zoomed out on phone.

```html
<link rel="stylesheet" href="style.css">
```
Connects your CSS file. Must be in `<head>` so styles load before page appears.

```html
<script src="script.js"></script>
```
Connects your JS file. Always at **bottom of body** — so HTML loads first, then JS runs on already-existing elements.

---

### Body Structure

```html
<header>
    <h1>chatbot-GenAI</h1>
</header>
```
- `<header>` → top section of page, semantic HTML tag
- `<h1>` → main title, biggest heading, only one per page
- No `id` needed here — JS doesn't need to find this element

---

```html
<div id="chat-box">
    <div class="message bot">
        Hi! Ask me anything!
    </div>
</div>
```

- `<div id="chat-box">` → the container where ALL messages appear
- `id="chat-box"` → unique name so JS can find it with `querySelector("#chat-box")`
- Inner `<div class="message bot">` → the welcome message shown on load
- `class="message bot"` → TWO classes on same element:
  - `message` → general styling for all messages
  - `bot` → specific styling for bot messages (different color than user)
- JS will dynamically create more divs like this as conversation happens

---

```html
<div id="input-area">
    <input type="text" id="user-input" placeholder="Enter here">
    <button id="send-btn">Send</button>
</div>
```

- `<div id="input-area">` → wrapper that holds input and button together side by side
- `<input type="text">` → text box where user types their message
- `id="user-input"` → JS uses this to grab what user typed via `.value`
- `placeholder="Enter here"` → grey hint text shown when input is empty
- `<button id="send-btn">` → the send button
- `id="send-btn"` → JS uses this to listen for click event
